02 — 5 capas de protección · de afuera hacia adentro
01
Capa de Perímetro
Primera línea · filtra antes de llegar a la aplicación
CONFIDENCIALIDAD + DISPONIBILIDAD
🛡️ Cloud Armor Incluido en Cloud Run
WAF gestionado por Google. Filtra DDoS, SQL injection, XSS, path traversal antes de que el request llegue a Node.js. El portal público (formulario ciudadano) está completamente expuesto — Cloud Armor es la primera defensa.
DDoS L3/L4/L7 OWASP Top 10 Rate limiting por IP Geo-blocking
Bloquea: bots que intentan llenar el formulario masivamente para generar trabajo falso al DPO
🔒 HTTPS / TLS 1.3 obligatorio $0 · automático
Todo el tráfico cifrado en tránsito. Vercel gestiona el certificado del frontend, Cloud Run gestiona el del backend. Sin opción de HTTP plano. Crítico porque se transmiten RUTs en cada formulario.
TLS 1.3 HSTS Cert auto-renovado
Protege: RUT y email del ciudadano en tránsito entre browser → Cloud Run
🌐 CORS restringido en Cloud Run $0 · configuración código
Cloud Run solo acepta requests desde los dominios de Vercel autorizados (ALLOWED_ORIGINS en env var). Nadie puede llamar la API desde un dominio distinto, aunque tenga el endpoint.
Origin whitelist Preflight validation
Bloquea: sitio malicioso que intente hacer llamadas a la API usando cookies del DPO
02
Capa de Identidad y Acceso
Quién eres · qué puedes hacer · IAM + Firebase Auth + App Check
CONFIDENCIALIDAD
🔐 Firebase Authentication $0 · email/pass ilimitado
Emite JWT tokens firmados por Google para DPOs, auditores y equipo AlignData. Cloud Run verifica el token en cada request con Firebase Admin SDK. El ciudadano no necesita cuenta — se autentica por código de email temporal.
JWT RS256 Token expira 1h Refresh token Revocación instantánea
Protege: nadie sin cuenta válida puede leer ni escribir solicitudes, aunque conozca los endpoints
📱 Firebase App Check $0 · configurable
Verifica que el request viene de tu aplicación React real desplegada en Vercel, no de Postman, scripts Python o aplicaciones no autorizadas. Usa un attestation token ligado al dominio.
reCAPTCHA v3 App attestation Domain binding
Bloquea: atacante que obtiene el API key público de Firebase e intenta usarlo desde scripts propios
🎭 IAM Roles GCP $0 · nativo GCP
Service Account de Cloud Run tiene solo los permisos mínimos: leer/escribir Firestore, leer/escribir Storage del proyecto específico. Ningún humano tiene acceso directo a producción. Todo pasa por el código.
Principio mínimo privilegio Service Account por ambiente Sin acceso humano directo a prod
Protege: si un desarrollador es comprometido, no puede leer datos de producción directamente
03
Capa de Protección de Datos
Cifrado en reposo · en tránsito · separación PII · Firestore Security Rules
CONFIDENCIALIDAD + INTEGRIDAD
🔑 Cifrado en reposo — AES-256 $0 · automático GCP
Firestore y Cloud Storage cifran todos los datos automáticamente con AES-256 antes de escribir en disco. No requiere configuración. Los backups también están cifrados.
AES-256 automático Backups cifrados Keys gestionadas por Google
Protege: si alguien accede físicamente a un disco de GCP, los datos son ilegibles
📜 Firestore Security Rules $0 · configuración
Reglas declarativas que controlan quién lee/escribe qué colección. Aunque alguien bypass-ee Cloud Run y llegue directo a Firestore, las reglas impiden el acceso sin JWT válido con el rol correcto.
Reglas por colección Validación JWT en regla Segunda línea vs bypass
Defensa en profundidad: si Cloud Run es bypasseado, Firestore igual rechaza el acceso
🏗️ Separación PII por proyecto $0 · arquitectura
Cada cliente tiene su propio proyecto Firebase (clinicax-prod, hospitalxyz-prod). Un DPO de Clínica X con token válido no puede ver datos de Hospital XYZ porque apunta a un proyecto GCP completamente distinto.
Aislamiento físico por cliente Sin multi-tenant riesgoso Un proyecto = un cliente
Protege: filtración de datos entre clientes — el mayor riesgo en portales SaaS
🔗 URLs firmadas Storage (72h) $0 · Firebase SDK
Los archivos en Cloud Storage no tienen URL pública. Cloud Run genera URLs temporales firmadas de 72h para que el ciudadano descargue su respuesta. Pasado ese tiempo, el link es inválido.
Expiración 72h Firmada con service account No indexable por Google
Protege: que una URL de descarga sea compartida o indexada accidentalmente
04
Capa de Auditoría Inmutable
Toda acción registrada · no editable · exportable para APDP · Art. 14 bis Ley 21.719
INTEGRIDAD · EVIDENCIA LEGAL
📋 Cloud Audit Logs (GCP) 400 GB/mes gratis
Log de infraestructura gestionado por Google. Registra TODA operación sobre recursos GCP: quién accedió a Firestore, qué modificó en Storage, desde qué IP. No puede ser borrado por ningún usuario, ni administrador del proyecto.
Inmutable por diseño Admin Activity logs Data Access logs Exportable a BigQuery
Evidencia legal: si la APDP fiscaliza, este log prueba que los datos no fueron alterados
📝 Audit Log de aplicación (Firestore) $0 · dentro del free tier
Colección /auditLog en Firestore donde Node.js registra cada acción semántica: "DPO Gonzáles cambió estado de VALIDADA a EN_PROCESO a las 14:32 del 10/03/2026". Solo Cloud Run puede escribir — reglas Firestore bloquean escritura directa.
Actor + acción + timestamp Solo escritura via Cloud Run Lectura: DPO + Auditor
Complementa Cloud Audit Logs con semántica de negocio: "quién hizo qué en el proceso ARCO"
📦 Cloud Storage Object Versioning $0.026/GB versiones antiguas
Cada archivo subido mantiene su historial de versiones. Si un DPO sobreescribe accidentalmente una evidencia, la versión anterior no desaparece. Para archivos de respuesta al titular esto es integridad documental.
Versiones históricas Soft delete configurable Restauración en 1 click
Protege: sobreescritura accidental o maliciosa de documentos de respuesta al titular
05
Capa de Gestión de Secretos
Credenciales fuera del código · rotación · acceso auditado · Secret Manager
CONFIDENCIALIDAD INTERNA
🗝️ Secret Manager GCP $0.06 / 10K accesos
Todas las credenciales (Resend API key, tokens de integración, claves de cifrado custom) viven en Secret Manager. Cloud Run las consume en runtime. Ningún desarrollador necesita ver los secretos de producción. Cada acceso queda logueado.
Resend API key Rotación automática Acceso logueado Versionado de secretos
Protege: que un desarrollador que deja el equipo siga teniendo credenciales válidas de producción
🤖 Service Account sin clave descargable $0 · best practice GCP
Cloud Run usa Workload Identity Federation — la Service Account está ligada al container, sin archivo JSON de clave que alguien pueda robar. La autenticación con Firebase Admin SDK es automática porque ambos están en GCP.
Workload Identity Sin JSON key files Auto-auth en GCP
Elimina el riesgo número 1 de fuga: el archivo credentials.json subido accidentalmente a GitHub
🔍 Security Command Center Tier Standard gratis
Monitoreo continuo de la postura de seguridad del proyecto GCP. Alerta si hay un bucket de Storage público accidentalmente, reglas de Firestore demasiado permisivas, o credenciales expuestas en código.
Misconfiguration detection Vulnerability scanning Alertas automáticas
Detecta: errores de configuración antes de que sean explotados
03 — qué pasa si algo falla · defensa en profundidad
🕵️
Atacante externo intenta leer datos
1.
Intenta hacer scraping del formulario público masivamente Bot enviando 1.000 solicitudes falsas por minuto ✓ Bloqueado: Cloud Armor rate limiting por IP
2.
Obtiene el VITE_FIREBASE_API_KEY del frontend (es público) Intenta leer directamente Firestore con ese key ✓ Bloqueado: Firestore Security Rules requieren JWT válido
3.
Obtiene un JWT válido phisheando a un DPO Intenta leer datos de otro cliente con el token ✓ Bloqueado: proyectos Firebase separados por cliente
👨‍💻
Desarrollador interno malintencionado
1.
Intenta acceder directamente a Firestore de producción Con su cuenta personal de Google ✓ Bloqueado: IAM — no tiene rol en el proyecto prod
2.
Busca credenciales en el repositorio de código Revisando commits, env vars, etc. ✓ Bloqueado: Secret Manager — no hay credenciales en el código
3.
Modifica un Cloud Audit Log para cubrir sus huellas Intenta borrar evidencia de su acceso ✓ Imposible: Cloud Audit Logs son inmutables por diseño
🏛️
Fiscalización APDP sin previo aviso
1.
APDP pide evidencia de que el RUT del ciudadano nunca fue expuesto En los últimos 12 meses ✓ Capa 4: Cloud Audit Logs exportados a BigQuery — respuesta en minutos
2.
APDP pide probar que los logs no fueron alterados Integridad de la evidencia ✓ Capa 4: logs inmutables gestionados por Google, no por AlignData
3.
APDP verifica que los datos están en Chile Art. 27 Ley 21.719 ✓ Capa 3: proyecto GCP en southamerica-west1 (Santiago) — certificable
